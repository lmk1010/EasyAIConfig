# 配置编辑容量与架构调研报告

## 背景

当前 `配置编辑` 页面已经同时承载 `Codex`、`Claude Code`、`OpenCode`、`OpenClaw` 四套异构配置。随着工具数量、插件数量和配置复杂度继续增长，现有 UI 结构已经接近上限：顶部工具切换拥挤，主编辑区横向空间被固定代码侧栏占用，工具内部继续堆 section 也会越来越难用。

本报告目标：

- 说明当前配置编辑页面的结构与瓶颈
- 分析为什么现在“四个都快放不下了”
- 给出适合未来承载 Codex 插件 / 多 Provider / 多实例配置的 UI 方向
- 给出建议的改造优先级

---

## 一、当前实现现状

### 1. 顶部结构

配置编辑页顶部目前同时包含三类入口：

1. 工具切换 Tab：`Codex / Claude Code / OpenCode / OpenClaw`
2. 视图切换：`仅表单 / 分栏 / 仅代码`
3. 右侧动作：`配置商店`，以及 OpenClaw 特有的 `最小 / 完整`

对应位置：

- `public/index.html:680`
- `public/index.html:721`
- `public/index.html:735`
- `public/index.html:740`

这意味着顶部不是单一层级，而是把“工具维度”“视图维度”“特殊模式维度”“配置来源入口”都塞到一行里。

### 2. 主体结构

主体区域是固定双栏：

- 左边：表单编辑区
- 右边：原始配置代码区

默认 `split` 模式下右侧会固定占据一列宽度，关键样式：

- `public/styles.css:9159`
- `public/styles.css:9183`

当前实现下：

- 常规布局：`1fr + 340/380px`
- 仅表单：退化成单栏
- 仅代码：退化成单栏

### 3. 每个工具是独立手写的一整套编辑区

页面并不是一个统一 schema renderer，而是四套并排的 DOM：

- `public/index.html:763` `data-tool-editor="codex"`
- `public/index.html:988` `data-tool-editor="claudecode"`
- `public/index.html:1086` `data-tool-editor="opencode"`
- `public/index.html:1328` `data-tool-editor="openclaw"`

右侧原始代码区同样也是按工具拆分的独立 section。

切换逻辑靠 JS 直接 show/hide：

- `public/app.js:10292`

这说明：

- 新增一个工具，需要补 HTML + CSS + JS 分支
- 新增一个工具内子能力（例如 Codex 插件），也大概率会继续在当前工具 DOM 里追加 section
- 复用能力弱，后续扩展成本会越来越高

### 4. 工具内部结构仍然是“section 堆叠”

以 Codex 为例，已经拆成多个 `details.cfg-section`：

- 模型与推理
- 行为与审批
- 上下文与性能
- 路径与环境
- 会话恢复
- Provider 与备份
- 开关选项
- 指令

关键位置：

- `public/index.html:768`
- `public/index.html:797`
- `public/index.html:830`
- `public/index.html:873`
- `public/index.html:904`
- `public/index.html:927`
- `public/index.html:955`
- `public/index.html:974`

这套模式在“工具少、字段中等”的时候有效，但继续增长时问题明显：

- section 越来越多
- 搜索能过滤字段，但不能改善整体导航结构
- 多实例配置（如插件列表、Provider 列表）不适合仅靠折叠 section 表达

### 5. 现在已经有“复杂度溢出”的早期征兆

OpenClaw 已经长出额外一层 `最小 / 完整` 模式：

- `public/index.html:735`
- `public/app.js:10338`

这实际上已经说明一个事实：

> 单靠“工具 tab + 一大块表单”已经装不下复杂工具了。

## 二、为什么现在四个就快放不下了

### 1. 顶层 Tab 承担了过多职责

当前顶部 tab 代表的是“工具产品边界”。

但未来真实的信息架构，不只是工具名：

- Codex：基础配置、模型、Provider、会话、插件、实验项、通知、代理、Profiles
- Claude Code：模型、认证、Projects、Hooks、缓存、权限策略
- OpenCode：Provider、Agent、IDE 集成、Auth、Gateway、Workspace
- OpenClaw：Gateway、Agent、Channel、Provider、Dashboard、安全、Webhook

如果把这些继续往顶层 tab 或一级折叠里塞，会出现：

- 顶部横向放不下
- 工具内部纵向找不到
- 同类能力（比如 Provider）无法跨工具复用交互模式

### 2. 右侧原始代码栏长期占宽

默认分栏模式下右侧固定吃掉 340~380px。

对于表单编辑来说，这会带来两个连锁问题：

- 左侧内容区宽度下降
- 表单为了适配宽度，会更早换行、更早变高

换句话说，**当前不是只有“项太多”的问题，而是“可用主编辑宽度太少”**。

### 3. 每个工具内部继续靠 section 堆叠，不适合异构配置

Section 适合：

- 基础选项
- 同构字段集合
- 表单字段较稳定的配置

Section 不适合：

- 动态数量的 Provider
- 插件列表 / 插件实例
- Profiles / 多环境预设
- IDE 集成目标（Cursor / VS Code / Windsurf）
- 安装型集成 / 外部能力接入

Codex 未来如果要加入插件配置，只靠继续塞 `details.cfg-section` 基本会失控。

### 4. 搜索只是“过滤器”，不是“导航器”

现在搜索逻辑本质是：

- 输入关键词
- 把不匹配的 section / field 隐藏

对应逻辑：

- `public/app.js:10268`

这能缓解“字段太多”，但无法解决：

- 我不知道该去哪个区块找
- 插件属于哪个层级
- 多实例对象之间怎么切换
- 工具内不同配置类型之间怎么跳转

### 5. 当前是“4 套手写页面塞进一个壳子”

这套结构的最大问题不是现在难看，而是后续不可持续：

- 工具越多，维护越重
- 工具越复杂，页面结构越分裂
- 一旦引入插件/Provider 列表/多实例，HTML 将迅速膨胀

## 三、未来最需要支持的配置类型

如果站在未来设计角度，配置编辑器至少要能容纳以下几类配置：

### 1. 固定基础配置

典型特征：字段数量有限，结构稳定。

例如：

- 默认模型
- 推理强度
- 沙箱模式
- 审批模式
- SQLite 路径
- 通知命令

这类最适合表单 section。

### 2. 可重复实例配置

典型特征：数量不定，用户会新增/删除/切换。

例如：

- Providers
- 插件实例
- Auth entries
- Profiles
- Workspace presets
- Gateway routes

这类不适合“平铺 section”，更适合列表 + 详情。

### 3. 安装 / 集成型配置

典型特征：不是纯配置文件字段，而是系统状态或外部目标。

例如：

- VS Code / Cursor / Windsurf 插件接入
- OpenCode Desktop
- Codex App
- Webhook / GitHub Actions / GitLab CI 集成

这类更像“资源节点”或“能力卡片”，不适合混在纯配置表单里。

### 4. 高级原始块 / JSON/TOML 片段

典型特征：用户需要直接编辑原始文本或 patch。

例如：

- OpenCode JSON 分片
- OpenClaw 某些复杂高级配置
- Codex 实验项

这类不适合强行全部表单化，应支持“高级原始配置节点”。

## 四、建议的 UI 方向

## 方案 A：工具一级 + 工具内二级导航

### 结构

- 顶部保留 4 个工具 tab
- 工具进入后，在左侧增加二级导航
- 主编辑区只展示当前二级节点
- 右侧原始配置默认折叠或抽屉化

### 示例

#### Codex

- 基础
- 模型与推理
- Provider
- 插件
- 会话与存储
- 安全与审批
- 高级
- 原始配置

#### Claude Code

- 基础
- 模型与行为
- 认证与账户
- Projects
- 缓存与成本
- 高级
- 原始配置

#### OpenCode

- 基础
- Provider
- Agent
- IDE 集成
- Auth
- 高级 JSON
- 原始配置

#### OpenClaw

- 基础
- Provider
- Agent
- Gateway
- Channel
- Dashboard
- 安全
- 高级
- 原始配置

### 优点

- 顶部工具数量可控
- 工具内部复杂度被拆开
- 用户心智更稳定
- 是对现有代码侵入最小的一步

### 缺点

- 还没有彻底解决 schema 复用问题
- 实现上仍然会保留部分工具特化逻辑

### 结论

这是最适合先做的短中期方案。

---

## 方案 B：列表 + 详情（特别适合 Provider / 插件 / 多实例）

### 结构

某些二级导航页不再直接平铺字段，而是：

- 左：实例列表
- 中：实例详情表单
- 右：原始片段 / 调试信息（可选）

### 适用场景

- Provider 管理
- 插件配置
- IDE 集成目标
- 多环境 profile
- Auth 凭证

### 为什么重要

Codex 插件未来如果有多个，绝不能继续做成：

- 插件 A 一个 section
- 插件 B 一个 section
- 插件 C 一个 section

正确方式应该是：

- 插件列表（左）
- 选中某插件后编辑（中）
- 高级配置或原始 JSON（右/抽屉）

### 结论

这个方案不需要整个配置编辑器都改成 master-detail，
但对“多实例对象”必须这么做。

---

## 方案 C：原始代码栏改为抽屉 / 底部面板，不再常驻分栏

### 当前问题

右侧代码栏长期占据 340~380px，直接压缩主表单空间。

### 建议做法

默认模式改为：

- 表单优先
- 原始配置通过按钮打开右侧抽屉或底部面板
- 保留 code-only 模式
- split 作为高级视图，不作为默认

### 优点

- 立刻给主区释放大量宽度
- 改动成本较低
- 与当前 `form / split / code` 模式天然兼容

### 结论

这属于低成本高收益，建议与方案 A 一起作为第一阶段落地。

---

## 方案 D：Schema / Manifest 驱动编辑器（中长期）

### 思路

将配置编辑从“手写 HTML”升级为“声明式 schema 渲染”：

```ts
{
  tool: 'codex',
  sections: [
    {
      id: 'model',
      title: '模型与推理',
      fields: [
        { key: 'model', type: 'select' },
        { key: 'reasoning_effort', type: 'select' }
      ]
    }
  ]
}
```

### 能解决什么

- 工具内结构统一
- 可做动态 section
- 可做插件注入
- 可做字段搜索、导航、权限控制
- 可做跨工具复用能力（如 Provider）

### 缺点

- 改造成本高
- 需要抽离 populate/save 逻辑
- 不适合第一阶段直接重构全量

### 结论

这是长期正确方向，但不建议第一刀全量重写。

## 五、推荐的落地顺序

## 第一阶段（P0）

### 目标

先把“放不下”和“继续扩展会炸”的问题稳住。

### 建议动作

1. 顶部只保留工具 tab
2. 工具内新增二级导航
3. 原始配置改为抽屉或非默认 split
4. 保留现有字段和 save/populate 逻辑

### 收益

- 4 个工具继续增长时还能撑住
- Codex 插件有位置可放
- OpenClaw 不再需要用“最小/完整”硬顶复杂度

## 第二阶段（P1）

### 目标

把“多实例对象”从 section 平铺中抽出来。

### 建议动作

1. Provider 改成列表 + 详情
2. 插件改成列表 + 详情
3. IDE 集成改成资源卡列表 + 详情

### 收益

- 多实例配置可扩展
- 用户能管理多个对象而不是在一个长表单里找

## 第三阶段（P2）

### 目标

开始收敛成统一的配置编辑平台。

### 建议动作

1. 抽公共 schema
2. 把 section / field renderer 通用化
3. 支持插件/工具注册式扩展

---

## 六、针对 Codex 插件的具体建议

如果 Codex 后续要支持“插件”，不建议把插件继续塞进当前 Codex 大表单底部。

建议结构：

- 工具：Codex
  - 基础
  - 模型与推理
  - Provider
  - 插件
  - 会话与存储
  - 高级
  - 原始配置

其中 `插件` 页采用：

- 左：已启用插件列表
- 中：选中插件配置表单
- 右：该插件原始配置（可抽屉）

插件页内应支持：

- 启用 / 禁用
- 排序
- 新增插件实例
- 删除插件实例
- 插件级高级配置

这样后续即便插件数量从 4 个变成 20 个，也不需要再改顶层结构。

## 七、当前结论

一句话总结：

> 当前配置编辑器已经不是“字段有点多”，而是信息架构层级不够。

根因不是单纯空间不够，而是：

- 顶部承载的维度太多
- 工具内部继续靠 section 堆叠
- 多实例与异构配置没有合适容器
- 固定原始代码栏长期占宽
- 结构仍是 4 套手写页面拼在一起

最推荐的演进路线：

1. 先做“工具一级 + 工具内二级导航”
2. 同时把原始代码栏改成抽屉 / 非默认 split
3. 对 Provider / 插件 / 集成改成列表 + 详情
4. 最后再逐步走向 schema 驱动
